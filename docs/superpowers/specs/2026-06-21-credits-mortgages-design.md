# Кредиты и Ипотека — настраиваемые разделы — Design

**Дата:** 2026-06-21
**Статус:** визуальный концепт утверждён, ключевые решения по объёму приняты, ревьюер спека — Approved; ожидает финального ОК пользователя перед планом реализации

## Цель

Добавить в SubEasy два новых раздела учёта — **Кредиты** (потреб, авто, рассрочки, долги) и **Ипотека** — так, чтобы:

1. Они выглядели **единым целым** с существующим трекером подписок (та же тема, карточки, шторки, синк).
2. По умолчанию приложение выглядело **ровно как сейчас** — только «Подписки», без новых переключателей. Существующий пользователь ничего не замечает.
3. Пользователь **сам включал** «Кредиты» и/или «Ипотеку» в настройках. Включил — раздел появился; выключил — исчез.
4. Кредиты/ипотека приносили то, чего у подписок не было: **остаток долга, ставку, ежемесячный платёж, прогресс выплаты, дату погашения и полный график платежей (амортизацию) — для обоих типов**.

**Решения по объёму (утв. пользователем 2026-06-21):** полный график платежей делаем **и у кредитов, и у ипотеки**; поддерживаем **обе схемы — аннуитетную и дифференцированную** — уже в v1.

Визуальный концепт реализован и согласован в **отдельном статичном HTML-макете** (вне git-репозитория): `SubEasy/mockups/credits-concept/index.html` — на уровень выше `neonsub/`, открывается локально на :4599. В главной **светлой теме** `claude`. Макет — референс вида, в репозиторий не коммитится.

## Контекст / текущее состояние

- Ядро данных — единственный тип `Subscription` ([src/lib/types.ts](../../../src/lib/types.ts)): `id, name, price, currency, category, cycle, cycleAnchor?, nextPaymentDate, startDate, paymentMethod, notes, color, icon, isActive, managementUrl?, createdAt, updatedAt, workspaceId?`.
- Хранение **offline-first**: каждый массив сущностей кладётся целым JSON-блобом в localStorage через generic-хук `useLocalStorage<T>(key, initial)` ([src/hooks/useLocalStorage.ts](../../../src/hooks/useLocalStorage.ts)); ключи `neonsub-subscriptions`, `neonsub-categories`, `neonsub-settings`. CRUD — в `useSubscriptions` / `useSettings`.
- Синк в Supabase — слой поверх localStorage ([src/lib/sync.ts](../../../src/lib/sync.ts)). Слияние — **уже generic**: `mergePersonalSubscriptionsForSync<T>` ([src/lib/syncMerge.ts](../../../src/lib/syncMerge.ts)) трогает только `T.id`/`T.workspaceId`. Маппинг snake_case↔camelCase — в [src/lib/dbMappers.ts](../../../src/lib/dbMappers.ts).
- Навигация — 4 нижних таба (Главная/Аналитика/Календарь/Настройки) + центральный «+» FAB ([src/components/layout/TabBar.tsx](../../../src/components/layout/TabBar.tsx)), `TabId`. Активный таб и ленивый рендер — в [src/app/page.tsx](../../../src/app/page.tsx) (AnimatePresence по `activeTab`). На 430px помещается ~5 табов, 6 ломают `min-w-[56px]`.
- Карточки/формы: `SubCard` (swipe + hold-delete), `SubList`, `SubDetail` (count-up, «оплачено»), `SubForm` (~1300 строк — пикеры валюты/цикла/даты/категории, кодек способа оплаты, OCR), общий `Modal` (draggable bottom-sheet), `Badge`/`Button` ([src/components/subscription/](../../../src/components/subscription/), [src/components/ui/](../../../src/components/ui/)).
- Аналитика суммирует `getMonthlyPrice()` ([src/components/analytics/](../../../src/components/analytics/)); календарь раскладывает `nextPaymentDate` ([src/components/calendar/](../../../src/components/calendar/)); напоминания — `getDaysUntilPayment`/`isPaymentSoon`. ID уведомления = `${subscriptionId}-soon-${date}`.
- Тема по умолчанию — **`claude` (светлая)**: `#faf9f5`, акцент `#d97757`, Lora+Inter ([src/lib/themes.ts](../../../src/lib/themes.ts), `DEFAULT_THEME = 'claude'`). Цвета задаются токенами в `globals.css` — компоненты не хардкодят цвет.
- i18n — словарь `translations.ts`, языки RU/EN ([src/lib/translations.ts](../../../src/lib/translations.ts)).
- Монетизация выключена — новые разделы **бесплатны**, без PRO-гейтов.

## Архитектурное решение

### 1. Модель данных — единый `Obligation` с дискриминатором `kind`

Расширяем существующий тип единым «обязательством» вместо трёх параллельных сущностей:

```ts
type ObligationKind = 'subscription' | 'credit' | 'mortgage';

interface Obligation {
  // — общие (есть сейчас) —
  id; name; currency; category?; nextPaymentDate; startDate?;
  paymentMethod?; notes?; color; icon?; isActive;
  createdAt; updatedAt; workspaceId?;
  kind: ObligationKind;            // НОВОЕ, default 'subscription'

  // — подписочные (как сейчас) —
  price?; cycle?; cycleAnchor?;    // для kind==='subscription'

  // — кредит/ипотека —
  lender?;            // банк / кредитор (текст; лого — монограмма)
  loanType?;          // 'consumer'|'auto'|'installment'|'debt'|'mortgage'
  principalAmount?;   // сумма кредита (исходная)
  outstandingBalance?;// остаток долга  ← заголовочное число
  interestRate?;      // ставка % годовых (опц.; 0 = рассрочка)
  monthlyPayment?;    // ежемесячный платёж  ← идёт в «платёж в месяц»
  paymentScheme?;     // 'annuity'|'differentiated' (default 'annuity')
  termMonths?;        // срок
  // — только ипотека —
  propertyName?;      // объект недвижимости
}
```

**Почему единый тип, а не три:** generic `mergePersonalSubscriptionsForSync<T>` работает без изменений; календарь/напоминания/синк — один конвейер; новый раздел почти бесплатен. Подписко-специфичные хелперы (`getMonthlyPrice`, `cycleAnchor`) защищаются проверкой `kind`.

`Subscription` остаётся как сужение `Obligation & { kind: 'subscription' }` для обратной совместимости существующего кода. **Важно для типизации:** поля, которые у `Subscription` обязательны (`price: number`, `cycle: BillingCycle`, `category: string`), в `Obligation` становятся опциональными (`price?`, `cycle?`, `category?`) — намеренное ослабление; сужение `Obligation & { kind:'subscription' }` должно требовать их обратно (через отдельный тип-гард или фабрику), иначе существующий код, читающий `sub.price`/`sub.cycle` напрямую, перестанет компилироваться.

### 2. Хранение и синк

- **localStorage:** один ключ `neonsub-subscriptions` хранит все `kind` (миграция при чтении: строки без `kind` получают `'subscription'`). Хуки `useCredits()`/`useMortgages()` — тонкие обёртки над тем же массивом с фильтром по `kind` (или один `useObligations(kind?)`).
- **Supabase:** **аддитивная** миграция (паттерн `supabase/2026-06-14-add-rates-column.sql` — миграции лежат плоскими дат-файлами прямо в `supabase/`, без подпапки `migrations/`; применяются вручную) — `ALTER TABLE subscriptions ADD COLUMN kind text DEFAULT 'subscription'` + nullable-колонки кредита (`lender, loan_type, principal_amount, outstanding_balance, interest_rate, monthly_payment, payment_scheme, term_months, property_name`). Существующие строки берут `'subscription'` по умолчанию. RLS не меняется (`user_id = auth.uid()` покрывает все `kind`). Расширить `dbMappers.ts` (`dbToObligation`/обратно).
- **Попутный долг (must-fix):** `cycleAnchor` есть в типе, но **не маппится** в `dbMappers.ts` — добавить колонку `cycle_anchor` в этой же миграции.

### 3. Навигация — настраиваемые пилюли-режимы (вариант B, без «Всё»)

- Нижний таб-бар **не трогаем** — остаётся 4 таба (`TabBar.tsx` без структурных изменений).
- Новое состояние `enabledSections` в настройках: `{ subscriptions: true, credits: false, mortgages: false }` (дефолт). Хранится в `AppSettings`, синкается как остальные настройки.
- Над списком на Главной (и в Аналитике/Календаре) — ряд **пилюль только из включённых разделов**. Порядок: Подписки · Кредиты · Ипотека.
  - Включён **1 раздел** → пилюль нет вообще (экран как сейчас).
  - Включено **≥2** → пилюли видны; активная задаёт «линзу» для Главной/Аналитики/Календаря.
- **Никакой объединённой ленты «Всё»** — по требованию пользователя удалена из концепта.
- Минимум **один** раздел всегда включён (нельзя выключить последний).
- FAB «+» создаёт сущность **активного режима** (в режиме «Кредиты» открывает форму кредита).
- Новый компонент `ModeSwitch` (пилюли). Состояние `activeMode` рядом с `activeTab` в `page.tsx`; при выключении раздела `activeMode` схлопывается на первый доступный.

### 4. Карточки и формы — переиспользуем

- **Карточка обязательства:** `SubCard` расширяется ветками по `kind`. Кредит/ипотека: монограмма банка (цвет бренда), бейдж типа («Кредит»/«Ипотека»), **заголовок = остаток долга**, платёж + дата, прогресс-бар «выплачено %». Подписка — без изменений.
- **Форма:** `SubForm` режимная по `kind`. Кредит — минимум **3 поля** (остаток + платёж + дата), остальное опционально (банк, ставка, сумма, срок, схема). Ипотека = поля кредита + объект недвижимости. Переиспользуем пикеры валюты/даты, `Modal`, `Button`.
- **Форма — схема платежа:** переключатель `paymentScheme` (аннуитет / дифференцированный) есть и у кредита, и у ипотеки; по умолчанию аннуитет.
- **Деталь:** новый `LoanDetail` (брат `SubDetail`) в том же `Modal`, **общий для кредитов и ипотеки**: донат «выплачено %», остаток/платёж/ставка/переплата/срок, **полный график платежей (амортизация)** свёрнутый под деталью, кнопка «Внести досрочный платёж».

### 5. Математика — новый модуль `src/lib/loanUtils.ts`

Чистый, без сайд-эффектов, покрыт юнит-тестами по образцу `src/lib/syncMerge.test.mjs` (hand-rolled `node:test` + `jiti` для загрузки `.ts`). В проекте **нет** `npm test`/jest/vitest — запуск вручную: `node src/lib/loanUtils.test.mjs`. Юнит-тесты важны: это единственная «новая математика».

Две ветки графика, выбор по `paymentScheme`:

- **Аннуитет** (`'annuity'`, дефолт): `M = P·r·(1+r)ⁿ / ((1+r)ⁿ − 1)`, `r = ставка/12`, `n = срок в мес`. Разрешён ручной ввод `monthlyPayment` (банковское округление). Проход: для каждого месяца `interest = balance·r; principal = M − interest; balance −= principal`.
- **Дифференцированный** (`'differentiated'`): тело гасится равными долями `principalPart = остаток / оставшийся_срок`; платёж месяца = `principalPart + остаток·r` (**убывает** каждый месяц). Проход: `payment_k = principalPart + balance·r; balance −= principalPart`.
- **Один проход амортизации** (любой ветки) даёт сразу всё: **график платежей** (дата · платёж · тело · проценты · остаток), **дату погашения**, **переплату** (Σ процентов), прогресс `(сумма − остаток)/сумма`, «осталось N платежей».
- **Досрочный платёж** идёт 100% в тело; повторный прогон показывает экономию срока/процентов.
- Тесты покрывают **обе схемы** (аннуитет, дифференц.) + кейс «0% рассрочка» (`r=0` → платёж = остаток/срок, без процентов).

### 6. Тема и стиль

- Главная тема — **светлая `claude`**; новые компоненты используют **только токены** (`bg-surface-2`, `text-neon`→акцент, `border-border-subtle`, `--shadow-card`), поэтому автоматически работают и в тёмных темах (green/purple/blue). Никакого хардкода цвета.
- Бренды банков (Сбер/Тинькофф/Альфа/ВТБ) — только в **монограмме-иконке**, не в чроме приложения.

### 7. Напоминания, аналитика, календарь

- **ID уведомлений** получают префикс `kind`: `credit:${id}-soon-${date}` — иначе коллизия с подписками. Платёж по кредиту/ипотеке генерит напоминание тем же конвейером.
- **Аналитика/Календарь** становятся «режим-зависимыми»: суммируют/раскладывают `monthlyPayment` для кредитов так же, как `getMonthlyPrice` для подписок; фильтр по `activeMode`. Сумма по кредитам = «платёж в месяц», **не** мешать с остатком долга в одно число.

## Порядок реализации (фазы)

1. **Фундамент:** тип `Obligation` + `kind`; миграция Supabase (`kind`, `cycle_anchor`, поля кредита); `dbMappers`; миграция чтения localStorage; `loanUtils.ts` с **обеими схемами** (аннуитет + дифференц.) + тесты (добавить `jiti` в `devDependencies` явно — сейчас резолвится транзитивно). Подписки работают как раньше.
2. **Настройка разделов:** `enabledSections` в `AppSettings` + синк; экран «Настройки → Разделы» с тумблерами; `ModeSwitch` (пилюли) на Главной; `activeMode` в `page.tsx`. Кредиты/ипотека ещё пустые.
3. **Кредиты + график:** форма (3 обязательных поля + переключатель схемы аннуитет/дифференц.), карточка с остатком долга, бейдж типа, FAB-режим, напоминания, **общий `LoanDetail` с полным графиком платежей** (обе схемы).
4. **Ипотека:** объект недвижимости (переиспользует форму/деталь/график кредита) + кнопка «досрочный платёж» (what-if) на том же графике.
5. **Полировка:** аналитика/календарь режим-зависимые; i18n RU/EN; пустые состояния разделов.

## Не делаем (YAGNI)

- Объединённую ленту «Всё» (удалена по требованию).
- Стратегии Snowball/Avalanche, геймификацию/бейджи-вехи.
- Escrow/страховку-налог отдельной строкой в ипотеке (трактуем ипотеку как долгосрочный кредит; вынести в будущий апдейт, если попросят).
- Шеринг кредитов/ипотеки в Family-workspace (сначала personal-only).
- Лого банков картинками (монограмма достаточно), OCR для кредитов, biweekly/ARM, снапшот курса для многолетних сумм (задокументировать поведение live-курса).
